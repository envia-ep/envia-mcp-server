# Envia MCP Server — Audit 2026-04-16

**Scope:** 90 tools, 1202 tests, ~150 source files.
**Method:** 3 parallel specialized agents (architecture, agent ergonomics, production readiness) + direct verification of 7 foundational files.
**Verdict:** Mid-tier solid. Gap to "world-class" is clear and bounded — ~2 focused sprints.

## What IS already world-class (keep doing)

1. **SSRF prevention** (`src/utils/api-client.ts:44-51`) — hardcoded `ALLOWED_HOSTS` allowlist blocks all non-Envia domains. Tests in `tests/security/ssrf.test.ts` verify.
2. **Retries with exponential backoff + jitter** for 429 and 5xx (`api-client.ts:95-145`). Smart.
3. **`friendlyError()`** sanitizes API errors, truncates messages to 200 chars, never leaks infra details (`api-client.ts:213-243`).
4. **Layer separation** — `types → services → tools → tests`. Easy to add new tools.
5. **Security test suite** — `tests/security/{auth,input-injection,resource-abuse,ssrf}.test.ts` exists and tests real scenarios.

## P0 — BLOCKERS (must fix before calling it production-ready)

### P0-1. HTTP mode has NO authentication layer

`src/index.ts:412-435`. Anyone who reaches `/mcp` uses the **server's default `ENVIA_API_KEY`** (set from env) unless they pass their own via tool param. Multi-tenant deployment → cross-contamination of API keys.

```ts
app.post('/mcp', async (req: Request, res: Response) => {
    const server = createEnviaServer();   // uses server-level ENVIA_API_KEY
    // ... NO auth header check anywhere
});
```

**Fix:** Require `Authorization: Bearer <MCP_SERVER_SECRET>` header on `/mcp`. Separate the server secret (client auth) from the Envia API key (upstream auth).

### P0-2. CORS `Access-Control-Allow-Origin: '*'`

`src/index.ts:401`. Any browser from any origin can hit `/mcp`. Combined with P0-1, this means `evil.com` can execute tools using the server's default API key.

**Fix:** `res.setHeader('Access-Control-Allow-Origin', process.env.ALLOWED_ORIGINS ?? 'https://claude.ai')`.

### P0-3. Zero observability

Only `console.error` at startup. In production: no way to debug "agent X says tool Y failed". No logger, no correlation IDs, no metrics, no tracing. SRE cannot operate this.

**Fix:** Add `pino` with request-scoped child loggers. Emit structured events on every tool call: `{ tool_name, api_key_hash, duration_ms, status, error_code }`.

### P0-4. Error map covers only 9 of ~30+ real Envia error codes

`src/utils/error-mapper.ts:23-69`. Covers 1101, 1105, 1115, 1125, 1127, 1129, 1140, 1220, 1282. In real sessions you've encountered 1147, 1149, 900, etc. — all fall through to the generic "Check the request parameters" suggestion.

**Fix:** Mine `log_errors` table in carriers DB for the top 30 codes by frequency. Write mappings for each. Target: 90%+ coverage of production errors.

### P0-5. `carrier: z.string()` across 20+ tools

Agents invent codes (`estafeta-global`, `fedex_mx`). No canonical list at the schema level.

**Evidence:** `src/tools/carriers-advanced/*.ts` all use `z.string().describe(...)`. `list-carriers` tool exists but agent has no hint to call it first.

**Fix:** Export `SUPPORTED_CARRIERS` const from `src/utils/schemas.ts` (seeded from carriers API). Replace 20+ occurrences of `z.string()` with `z.enum(SUPPORTED_CARRIERS)`. Optional refresh job weekly.

## P1 — WORLD-CLASS BLOCKERS

### P1-1. 111 `Record<string, unknown>` request bodies bypass Zod validation

Every `create-*` / `update-*` tool has:
```ts
const body: Record<string, unknown> = { name: args.name, ... };
await activeClient.post<T>(url, body);
```
Zod validates INPUT from agent, but not the body sent to Envia API. A typo in a key name fails silently.

**Fix:** Define typed payloads (`CreateClientPayload`, `UpdateWebhookPayload`, etc.) in `src/types/`. Compiler catches mistakes.

### P1-2. 15 unsafe `as` casts (several with `as unknown as`)

Worst offender: `src/tools/create-label.ts:912-913`. Double cast = no guardrails.

**Fix:** Type guards (`function isX(v: unknown): v is X`) + runtime validation at API boundary.

### P1-3. Inconsistent response pattern (6 tools raw, 82 via helper)

6 tools still use `return { content: [{ type: "text", text: ... }] }` directly. 82 use `textResponse()`. If MCP SDK response shape changes, the 6 break silently.

**Files:** `track-package.ts:64-65`, `cancel-shipment.ts:56-62`, and 4 others.

**Fix:** Migrate to `textResponse()`. Add ESLint rule `no-restricted-syntax` to block raw pattern.

### P1-4. Status IDs as magic numbers in descriptions

`list-shipments.ts`: `status_id: z.number().int().optional().describe('1=Created, 2=In transit...')`.

Agent must read the description (may not), then pass a number (may mistype). If API re-numbers, descriptions drift from reality.

**Fix:** Export `ShipmentStatus` enum from `src/types/shipments.ts`. Use `z.enum([1,2,3,...])` or `z.nativeEnum(ShipmentStatus)` in schemas. Reference in `.describe()`.

### P1-5. Tool pairs with ambiguous decision

- `envia_track_package` (public) vs `envia_track_authenticated` — agent sees both, no clear rule when to use which
- `envia_validate_address` vs `envia_locate_city` — both address-related, overlapping for Colombia
- `envia_get_shipping_rates` vs `envia_rate_order` — second dropped but mental model still ambiguous

**Fix:** Every ambiguous tool description opens with a decision line: *"Use this ONLY if [X]. For [default case], use [other tool] instead."*

### P1-6. Address builder duplication

`src/builders/address.ts`: 3 near-identical functions (`buildGenerateAddress`, `buildGenerateAddressFromLocation`, `buildGenerateAddressFromShippingAddress`), ~70% overlap. Bug in one won't fix the others.

**Fix:** Single polymorphic builder with source adapters.

### P1-7. No standardized pagination output

List tools vary: some print "Showing 20 of 150", others just print results. Agents iterating large datasets have no reliable signal.

**Fix:** Shared `formatPagination(page, total, shown)` helper, mandated in every list tool output.

### P1-8. Monolithic `createEnviaServer()` at 487 lines

`src/index.ts` has 100+ manual `registerX(server, client, config)` calls. Forgetting one silently omits the tool. Scales poorly past 120 tools.

**Fix:** Each barrel exports a `register(server, client, config)` function that registers all its tools. Main index loops through barrels. Or: tool registry pattern with `TOOL_REGISTRY.forEach(fn => fn(server, client, config))`.

## P2 — Polish

- **No Dockerfile / K8s manifests** — deployment story unclear
- **No health check endpoint** (`/health`, `/ready`)
- **No graceful shutdown** (SIGTERM handler)
- **No startup config validation** (fail-fast if `ENVIA_API_KEY` missing)
- **Magic `type: 1|2` in addresses** — should be `'origin'|'destination'` or `z.enum([1,2])` with clear docs
- **Irreversibility warnings in JSDoc but not in schema** (`fulfill-order.ts`)
- **CI workflow exists but no `npm audit`** in pipeline
- **`MCP_EXPANSION_PLAN.md` + `MCP_REMAINING_PHASES_GUIDE.md` + `AUDIT_2026_04_16.md`** — docs accumulating, consider consolidating

## Roadmap recomendado (3 sprints, ~3-4 semanas)

### Sprint 1 — "Make it safe to ship" (P0)
**Goal:** Flip MCP server from "single-tenant dev tool" to "deployable service"
- P0-1: HTTP Bearer auth layer (`MCP_SERVER_SECRET`)
- P0-2: CORS from env var
- P0-3: `pino` logger + request-scoped correlation IDs + tool-call events
- P0-4: Expand `ERROR_MAP` to 30+ codes (mine from `log_errors` table)
- P2: Dockerfile + health endpoint + graceful shutdown + startup validation
- **Exit criteria:** can be deployed publicly to staging, errors traceable, logs structured

### Sprint 2 — "Make agents 30% more reliable" (P0-5 + P1 type safety + ergonomics)
- P0-5: `SUPPORTED_CARRIERS` enum, apply in 20+ tools
- P1-4: Status ID enums across list tools
- P1-5: Disambiguate tool pairs (descriptions start with decision rules)
- P1-7: Standardized pagination output
- P1-3: Migrate 6 raw-response tools to `textResponse()` + ESLint rule
- **Exit criteria:** agent-ergonomics score 6.5→8.5

### Sprint 3 — "Internal quality" (P1 refactor)
- P1-1: Typed payloads for all mutation tools (kill `Record<string, unknown>` in bodies)
- P1-2: Type guards instead of `as unknown as`
- P1-6: Single address builder
- P1-8: Tool registry pattern (auto-register from barrels)
- Service-layer error-path tests
- **Exit criteria:** type-safety score 6→9

## Conclusión

**No es un proyecto mediocre.** La capa fundacional (api-client, tests de security, arquitectura por capas, Zod en inputs) está bien. Lo que falta NO es re-escribir, es **blindar la capa de operaciones** (auth HTTP, observabilidad, deployment) y **pulir la experiencia de agente** (enums, disambiguation, pagination).

La distancia a "Stripe-level DX" se mide en días, no meses. El plan de 3 sprints arriba te lleva ahí si lo ejecutas en orden.

Siguiente acción sugerida: empezar Sprint 1 con P0-3 (logger) + P0-1 (auth) en la misma sesión — son independientes y son los que más desbloquean todo lo demás.
